    For EVERY route categorized as "write" in ROUTE-INVENTORY.md, modify the route handler to follow this exact pattern:

    ```typescript
    import { NextResponse } from 'next/server'
    import { createClient } from '@/lib/supabase/server'
    import { relevantSchema } from '@/schemas/relevant-domain'

    export async function POST(request: Request) {
      // 1. Auth guard (SEC-02)
      const supabase = await createClient()
      const { data: { user }, error: authError } = await supabase.auth.getUser()

      if (authError || !user) {
        return NextResponse.json(
          { error: 'Unauthorized' },
          { status: 401 }
        )
      }

      // 2. Parse raw JSON body (SEC-07)
      let body: unknown
      try {
        body = await request.json()
      } catch {
        return NextResponse.json(
          { success: false, error: 'Invalid JSON body' },
          { status: 400 }
        )
      }

      // 3. Zod validation (SEC-07)
      const result = relevantSchema.safeParse(body)
      if (!result.success) {
        return NextResponse.json(
          { success: false, error: result.error.flatten() },
          { status: 400 }
        )
      }

      // 4. Use result.data (typed) for business logic
      // ... existing handler logic, replacing raw body access with result.data ...
    }
    ```

    Rules:
    - Auth guard ALWAYS comes first (before body parsing)
    - Use `safeParse()` NEVER `parse()` (safeParse does not throw)
    - Use `error.flatten()` NEVER `error.format()` (format() is deprecated in Zod v4)
    - Wrap `request.json()` in try/catch separately from Zod validation
    - The 401 response shape is `{ error: 'Unauthorized' }`
    - The 400 response shape for invalid JSON is `{ success: false, error: 'Invalid JSON body' }`
    - The 400 response shape for validation failure is `{ success: false, error: result.error.flatten() }`
    - Do NOT use `auth.getSession()` — it is insecure on the server (no auth-server revalidation)
    - Preserve all existing business logic below the guard+validation section
    - If the route has PATCH or DELETE handlers, apply the same auth guard pattern to those methods too
    - If PATCH/DELETE does not accept a body, skip the Zod validation but keep the auth guard

    For routes categorized as "read-authenticated" in ROUTE-INVENTORY.md:
    - Add only the auth guard (steps 1 above), no Zod validation needed for GET
    - Same 401 pattern: `if (authError || !user) { return NextResponse.json({ error: 'Unauthorized' }, { status: 401 }) }`
