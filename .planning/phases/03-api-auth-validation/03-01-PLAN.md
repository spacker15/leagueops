  <action>
    1. Discover every API route file in the project by scanning `src/app/api/**/route.ts` (or `app/api/**/route.ts` if no `src/` prefix).

    2. For each route file found, read it and categorize into one of four buckets:
       - **write**: Has POST, PATCH, PUT, or DELETE handler AND requires authentication
       - **read-authenticated**: Has GET handler AND requires authentication (accesses user-specific data)
       - **public**: Intentionally accessible without auth (e.g., /api/join, /api/checkins, /api/public-results)
       - **engine-trigger**: Engine/cron trigger routes (e.g., /api/engine/weather, /api/engine/referee)

    3. Create `.planning/phases/03-api-auth-validation/ROUTE-INVENTORY.md` with this format:
       ```markdown
       # Route Inventory — Phase 03

       Generated: [date]

       ## Classification Rules
       - write: POST/PATCH/PUT/DELETE + auth required → gets auth guard (SEC-02) + Zod validation (SEC-07)
       - read-authenticated: GET + auth required → gets auth guard (SEC-02)
       - public: No auth guard → gets documentation comment (SEC-02) + rate limiting (SEC-08)
       - engine-trigger: Internal trigger → gets auth guard (SEC-02) + rate limiting (SEC-08)

       ## Routes

       | Route | File Path | Methods | Category | Auth Guard | Zod Schema | Rate Limit |
       |-------|-----------|---------|----------|------------|------------|------------|
       | /api/example | src/app/api/example/route.ts | POST | write | YES | YES | NO |
       ```

    4. Create `src/schemas/index.ts` as a barrel file. For each write route discovered in step 2, create a corresponding Zod schema file in `src/schemas/[domain].ts` and re-export from the barrel.
       - Examine each write route's request body handling to determine the expected shape
       - Use `z.object({...})` with appropriate validators (z.string().uuid(), z.number().int().min(), z.string().datetime(), etc.)
       - Export both the schema and its inferred TypeScript type: `export type XxxInput = z.infer<typeof xxxSchema>`
       - The barrel file (`src/schemas/index.ts`) should re-export all schemas: `export { createScheduleSchema } from './schedule'` etc.
       - If a route does not accept a body (GET-only), it does not need a Zod schema
       - Import `z` from `zod` (Zod v4 syntax: `import { z } from 'zod'`)

    5. If no API routes exist yet (greenfield), create placeholder schemas for the domains mentioned in the feature backlog (schedules, programs, teams, registrations) with TODO comments noting they should be updated when routes are created.
  </action>
