2. After the `runWeatherEngine` call (line 42-47), BEFORE the `return NextResponse.json(result)`, add notification wiring. The weather engine returns a result object — check whether it has `result.alerts` array. If so, iterate and insert notifications:

   ```typescript
   const result = await runWeatherEngine(
     Number(complex_id),
     api_key ?? process.env.OPENWEATHER_API_KEY,
     Number(event_id),
     sb
   )

   // NOT-02: Wire weather alert notifications (per D-15, D-16, D-17)
   if (result.alerts && Array.isArray(result.alerts)) {
     for (const alert of result.alerts) {
       try {
         await insertNotification(Number(event_id), 'weather_alert', 'event', null, {
           title: `Weather Alert: ${alert.type ?? 'Unknown'}`,
           summary:
             alert.description ??
             alert.message ??
             `Weather alert detected at complex ${complex_id}`,
           detail: `Complex ${complex_id} — ${alert.type ?? 'weather'}: ${alert.description ?? alert.message ?? ''}`,
           cta_url: `${process.env.NEXT_PUBLIC_APP_URL ?? ''}?tab=weather`,
         })
       } catch (notifErr) {
         // Non-fatal: notification failure must not break weather engine response
         console.error('Failed to insert weather notification:', notifErr)
       }
     }
   }

   return NextResponse.json(result)
   ```

   The dedup_key is a STORED computed column in the notification_queue table (Phase 7) — it handles deduplication automatically at the DB level. The `event_id + alert_type + scope` combination ensures duplicate inserts within the 5-minute window are rejected by the DB constraint.
