**Part A: Append LAYER 4 to rls_migration.sql — Anon read policies for public tables (per D-08, D-09)**

Add anon SELECT policies for these 6 public tables ONLY:

```sql
-- ==========================================================
-- LAYER 4: Anon read policies (public tables per D-08, D-09)
-- Open SELECT, no event filter — public results site needs cross-event browsing
-- ==========================================================

CREATE POLICY "anon_select_events" ON events
  FOR SELECT TO anon
  USING (true);

CREATE POLICY "anon_select_event_dates" ON event_dates
  FOR SELECT TO anon
  USING (true);

CREATE POLICY "anon_select_fields" ON fields
  FOR SELECT TO anon
  USING (true);

CREATE POLICY "anon_select_teams" ON teams
  FOR SELECT TO anon
  USING (true);

CREATE POLICY "anon_select_games" ON games
  FOR SELECT TO anon
  USING (true);

CREATE POLICY "anon_select_registration_divisions" ON registration_divisions
  FOR SELECT TO anon
  USING (true);
```

No anon policies on any other table. Per D-02, sensitive tables (ops_alerts, ops_log, user_roles) have ZERO anon policies — Postgres RLS default-deny returns zero rows.

Also add `divisions` to public anon SELECT if a `divisions` table exists (check RESEARCH.md Group A — `registration_divisions` is listed, verify if there is a separate `divisions` table).

**Part B: Create supabase/rls_rollback.sql (per D-07)**

Generate a rollback script that:
1. Drops ALL new policies created by rls_migration.sql
2. Drops the user_event_ids() function
3. Recreates the original "Allow all" (or equivalent) policies for every table

Format:
```sql
-- ==========================================================
-- RLS ROLLBACK: Revert Phase 4 migration
-- Use ONLY if RLS migration causes production issues
-- Safe because Phase 3 auth guards protect API routes at app layer
-- ==========================================================

-- Step 1: Drop all new policies
DROP POLICY IF EXISTS "auth_select_events" ON events;
DROP POLICY IF EXISTS "auth_insert_events" ON events;
-- ... (every policy created in rls_migration.sql)

-- Step 2: Drop helper function
DROP FUNCTION IF EXISTS user_event_ids();

-- Step 3: Restore permissive policies
CREATE POLICY "Allow all" ON events FOR ALL USING (true) WITH CHECK (true);
-- ... (every table gets its original policy back)
```

Use the EXACT original policy names from the source SQL files (e.g., "schedule_rules_all", "Allow all on registration_invites", etc.).

Smoke Test 2 — Verify no "Allow all" policies remain (except on user_roles):
```sql
SELECT schemaname, tablename, policyname
FROM pg_policies
WHERE policyname LIKE '%Allow all%'
  OR policyname IN ('schedule_rules_all', 'weekly_overrides_all', 'schedule_audit_log_all')
ORDER BY tablename;
```
Expected: 0 rows (all permissive policies dropped). If user_roles policies appear, that's correct — they were not "Allow all".

Smoke Test 3 — Count new policies per table:
```sql
SELECT tablename, COUNT(*) as policy_count
FROM pg_policies
WHERE schemaname = 'public'
GROUP BY tablename
ORDER BY tablename;
```
Expected: Every table in the inventory has 1-5 policies. No table has 0 policies (except user_roles which keeps its existing ones).

Smoke Test 4 — Verify anon policies exist only on public tables:
```sql
SELECT tablename, policyname
FROM pg_policies
WHERE roles::text LIKE '%anon%'
  AND schemaname = 'public'
ORDER BY tablename;
```
Expected: Only events, event_dates, fields, teams, games, registration_divisions appear.

If any smoke test fails, check the error, fix the SQL in rls_migration.sql, and re-apply. Do NOT proceed until all 4 smoke tests pass.
