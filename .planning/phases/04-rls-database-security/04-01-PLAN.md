**LAYER 0: Deploy user_event_ids() helper function (per D-03, D-04, D-05)**

```sql
CREATE OR REPLACE FUNCTION user_event_ids()
RETURNS SETOF BIGINT
LANGUAGE sql
STABLE
SECURITY DEFINER
SET search_path = public
AS $$
  SELECT DISTINCT event_id
  FROM user_roles
  WHERE user_id = auth.uid()
    AND is_active = TRUE
    AND event_id IS NOT NULL;
$$;

GRANT EXECUTE ON FUNCTION user_event_ids() TO authenticated, anon;
```

**LAYER 2: Authenticated SELECT policies**

For each table, create the appropriate SELECT policy based on its group:

GROUP B — Direct event_id (per D-01, use `event_id IN (SELECT user_event_ids())`):
Tables: referees, volunteers, incidents, medical_incidents, weather_alerts, ops_log, complexes, operational_conflicts, weather_readings, lightning_events, heat_events, event_rules, rule_changes, schedule_snapshots, conflict_engine_runs, ops_alerts, shift_handoffs, schedule_rules, weekly_overrides, schedule_audit_log, division_hierarchy, eligibility_violations, multi_game_approvals, registration_questions, season_game_days, field_divisions, event_admins, player_qr_tokens, qr_checkin_log, portal_checkins, registration_invites, program_teams, team_registrations, field_blocks, players

Policy name pattern: `"auth_select_{table_name}"`
```sql
CREATE POLICY "auth_select_{table}" ON {table}
  FOR SELECT TO authenticated
  USING (event_id IN (SELECT user_event_ids()));
```

Note: `players` table has `event_id` added by player_eligibility.sql — use direct scoping.
Note: `field_blocks` has `event_id` added by phase4_migration.sql — use direct scoping.

GROUP A — Public tables also get authenticated SELECT (event-scoped for auth users):
Tables: events, event_dates, fields, teams, games, registration_divisions

For `events` table: the event_id IS the `id` column:
```sql
CREATE POLICY "auth_select_events" ON events
  FOR SELECT TO authenticated
  USING (id IN (SELECT user_event_ids()));
```

For other Group A tables: use `event_id IN (SELECT user_event_ids())`.

GROUP C — Indirect scope (EXISTS join pattern):
- `ref_assignments`: EXISTS (SELECT 1 FROM games g WHERE g.id = ref_assignments.game_id AND g.event_id IN (SELECT user_event_ids()))
- `vol_assignments`: EXISTS (SELECT 1 FROM games g WHERE g.id = vol_assignments.game_id AND g.event_id IN (SELECT user_event_ids()))
- `player_checkins`: EXISTS (SELECT 1 FROM games g WHERE g.id = player_checkins.game_id AND g.event_id IN (SELECT user_event_ids()))
- `referee_availability`: EXISTS (SELECT 1 FROM referees r WHERE r.id = referee_availability.referee_id AND r.event_id IN (SELECT user_event_ids()))
- `registration_answers`: EXISTS (SELECT 1 FROM team_registrations tr WHERE tr.id = registration_answers.team_reg_id AND tr.event_id IN (SELECT user_event_ids()))

GROUP E — Event-agnostic tables:
- `programs`: `FOR SELECT TO authenticated USING (true)` — all authenticated users can see all programs
- `program_leaders`: `FOR SELECT TO authenticated USING (true)` — all authenticated users can see program leaders
- `seasons`: `FOR SELECT TO authenticated USING (true)` — all authenticated users can see seasons

**LAYER 3: Authenticated write policies (INSERT, UPDATE, DELETE)**

For each GROUP B table, create INSERT, UPDATE, DELETE policies:
```sql
CREATE POLICY "auth_insert_{table}" ON {table}
  FOR INSERT TO authenticated
  WITH CHECK (event_id IN (SELECT user_event_ids()));

CREATE POLICY "auth_update_{table}" ON {table}
  FOR UPDATE TO authenticated
  USING (event_id IN (SELECT user_event_ids()))
  WITH CHECK (event_id IN (SELECT user_event_ids()));

CREATE POLICY "auth_delete_{table}" ON {table}
  FOR DELETE TO authenticated
  USING (event_id IN (SELECT user_event_ids()));
```

For GROUP A tables: same pattern (events uses `id` instead of `event_id` for writes).

For GROUP C tables: same EXISTS join pattern in WITH CHECK / USING.

For GROUP E tables:
- `programs`: INSERT open to authenticated (`WITH CHECK (true)`), UPDATE scoped to program leaders (`USING (EXISTS (SELECT 1 FROM program_leaders pl WHERE pl.program_id = programs.id AND pl.user_id = auth.uid()))`), DELETE same scope as UPDATE
- `program_leaders`: INSERT/UPDATE/DELETE open to authenticated (app-layer enforced)
- `seasons`: INSERT/UPDATE/DELETE open to authenticated (app-layer enforced per Phase 3 auth guards)

CRITICAL per D-02: `ops_alerts`, `ops_log`, `user_roles` get ZERO anon policies (handled in next plan). They DO get authenticated policies here.

Each policy must use exact naming convention: `auth_{operation}_{table_name}`.
